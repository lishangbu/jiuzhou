# 坊市部分购买设计

**目标**

支持坊市挂单在 `qty > 1` 时按自定义数量购买；当挂单已部分成交后，卖家下架剩余物品时，手续费按剩余未成交比例退还。

**核心方案**

1. 同一条 `market_listing` 挂单继续作为单一交易入口，不拆成多条虚拟挂单。
2. `market_listing` 新增 `original_qty` 字段，作为手续费比例退还的单一数据源。
3. 部分购买时仅结算本次购买数量，并把剩余数量继续保留在原挂单与原 `item_instance` 上。
4. 下架时按 `listing_fee_silver * qty / original_qty` 向下取整退还手续费。

**数据流**

1. 前端坊市列表点击购买。
2. 当挂单数量大于 1 时，前端弹出统一数量输入弹窗，收集 `buyQty`。
3. 路由 `POST /api/market/buy` 把 `listingId + qty` 传给 `marketService.buyMarketListing`。
4. 服务层校验购买数量、挂单状态、钱包余额后，按本次数量完成灵石结算、税费计算、邮件发货与成交记录。
5. 若未买完，则更新 `market_listing.qty` 与 `item_instance.qty` 为剩余数量；若买完，则沿用现有整单售罄逻辑。
6. 卖家下架时，服务层基于 `original_qty` 与当前剩余 `qty` 计算应退手续费，并返还银两。

**复用与抽象**

1. 后端抽取共享纯函数模块，统一处理购买数量规范化、部分购买价格计算、手续费比例退还，避免在创建/购买/下架链路重复写数量与金额规则。
2. 前端抽取坊市购买共享模块，统一处理可购买数量夹紧、总价计算、购买按钮文案与表单校验，避免桌面表格、移动卡片、移动预览抽屉各写一份判断。

**边界条件**

1. `qty <= 1` 的挂单保持一键购买体验，不额外增加输入步骤。
2. 购买数量必须是 `1 ~ row.qty` 的正整数，非法值统一被夹紧或拒绝。
3. 手续费退还使用向下取整，保证累计退还金额不超过原始手续费。
4. 部分购买后若剩余数量为 0，必须走售罄分支，不能保留一条 `qty = 0` 的活跃挂单。
5. 邮件附件与成交记录只记录本次实际购买数量，不能继续沿用挂单原始数量。

**非目标**

1. 不改造坊市挂单展示结构为多条子挂单。
2. 不在本次需求中引入新的税费策略或新的交易状态。
3. 不自动同步数据库结构；表结构同步仍由 Prisma `db push` 负责。
