# 角色功法散功设计

## 目标

为玩家角色已学习功法增加“散功”能力，用于主动舍弃一门未装配功法。该能力只处理角色功法链路，不返还任何资源、不返还功法书，也不扩展到伙伴或背包其它系统。

## 命名与主题

- 功能名称：`散功`
- 入口按钮：`散功`
- 确认标题：`确认散去「功法名」？`

选择 `散功` 的原因：

- 语义直接对应“主动散去已修功法”，不需要玩家额外学习成本。
- 题材贴合当前修仙世界观，比“遗忘”“删除”更自然。
- 能自然承接“散去道痕、不返资源”的风险提示文案。

## 范围

### 做什么

- 在角色功法面板的“已学功法”列表中，为未装配功法提供 `散功` 操作。
- 新增角色功法服务与路由入口，删除 `character_technique` 中对应未装配记录。
- 前端增加确认弹窗与操作文案。
- 增加服务端与前端回归测试，锁定规则与文案。

### 不做什么

- 不返还功法书、修炼资源、层数或任何补偿。
- 不支持对已装配功法直接执行散功。
- 不改动伙伴打书、背包用书、洞府研修等其它功法链路。
- 不新增兜底兼容分支，不为历史脏数据额外扩展 UI 能力。

## 规则

1. 仅未装配功法可散功。
2. 已装配功法点击入口时前端禁用，服务端再次拒绝，保证规则单一出口。
3. 散功成功后，该功法从角色已学列表移除。
4. 不返还任何资源或功法书。

## 架构设计

### 后端

- 路由层在 `server/src/routes/characterTechniqueRoutes.ts` 新增 `POST /api/character/:characterId/technique/:techniqueId/dissipate`。
- 服务层在 `server/src/services/characterTechniqueService.ts` 新增 `dissipateTechnique(characterId, techniqueId)`。
- 服务内部只做一件事：
  - 锁定目标功法行；
  - 判断是否存在；
  - 判断是否已装配；
  - 未装配则直接删除该行；
  - 返回统一成功/失败文案。

### 前端

- API 层在 `client/src/services/api/technique.ts` 增加 `dissipateCharacterTechnique` 请求封装。
- 功法弹窗在 `client/src/pages/Game/modules/TechniqueModal/index.tsx` 的“已学功法”列表增加 `散功` 按钮。
- 新增共享文案模块，集中维护：
  - 散功按钮状态；
  - 散功确认标题；
  - 散功确认提示文案。

## 数据流

1. 玩家在“已学功法”列表点击某门未装配功法的 `散功`。
2. 前端读取当前装配态，决定按钮是否可点。
3. 点击后弹出确认框，明确“不返还资源或功法书”。
4. 确认后调用角色功法散功接口。
5. 服务删除对应 `character_technique` 行并返回结果。
6. 前端刷新功法状态，列表移除该功法。

## 性能考虑

- 不新增列表二次遍历；前端继续复用现有 `equippedSlotByTechId` 索引结果判断按钮态。
- 不引入新的全局查询；服务端只锁定目标功法单行，不扫描整表。
- 不触发与战斗无关的额外刷新链路；因为仅允许删除未装配功法，不影响当前战斗属性、技能槽与在线战斗快照。
- 散功文案与按钮态抽到共享纯函数，避免 JSX 渲染期反复拼接多处分支。

## 边界条件

1. 已装配功法不可散功，服务端返回明确提示“请先取消运功”。
2. 功法不存在或不属于当前角色时返回“未学习该功法”。
3. 成功散功后若再次点击旧按钮，前端刷新后应自然消失，不保留陈旧状态。
4. 该能力只面向人物功法，不可外溢到伙伴功法或功法书道具层。
